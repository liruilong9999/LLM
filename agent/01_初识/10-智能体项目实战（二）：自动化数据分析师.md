# 智能体项目实战（二）：自动化数据分析师

## 本篇目标

本篇设计一个自动化数据分析师 Agent。它把自然语言问题转成数据查询、统计计算、图表和报告。

学完后，你应该能：

- 设计自然语言到 SQL(结构化查询语言) 的安全链路。
- 理解 code interpreter(代码解释器) 在数据分析中的作用。
- 规划从查询到报告的端到端流程。

## 先修知识

建议先读完工具调用、任务规划和可靠性基础。你需要知道 SQL、CSV(逗号分隔值文件) 和基本统计指标。

## 场景说明

用户输入：

```text
“帮我分析 3 月销售额下降的原因，并生成一份图表报告。”
```

数据分析 Agent 需要：

1. 理解分析目标。
2. 找到相关数据表或文件。
3. 生成安全查询。
4. 执行统计计算。
5. 绘制图表。
6. 解释变化原因。
7. 输出可复核报告。

## 总体流程

```mermaid
flowchart LR
    用户问题 --> 问题澄清
    问题澄清 --> 数据定位
    数据定位 --> 查询生成
    查询生成 --> 查询审核
    查询审核 --> 执行查询
    执行查询 --> 数据分析
    数据分析 --> 图表生成
    图表生成 --> 报告生成
    报告生成 --> 结果复核
```

## 核心模块

### 数据目录

Agent 需要知道有哪些表、字段和含义。不要让模型凭空猜字段名。

数据目录示例：

```text
表：orders
字段：
- order_id：订单编号
- city：城市
- amount：订单金额
- created_at：下单时间
- channel：销售渠道
- status：订单状态
```

数据目录不只是字段说明，还应该包含业务口径：

```text
指标：销售额
定义：已支付订单金额，不含退款金额
时间口径：按支付时间统计
过滤条件：排除测试订单和取消订单
负责人：数据团队
更新时间：2026-04-28
```

没有指标字典时，Agent 很容易把“下单金额”“支付金额”“净销售额”混为一谈。

### 查询生成

模型可以生成 SQL，但必须受约束：

- 只允许 SELECT(查询)。
- 禁止 DELETE(删除)、UPDATE(更新)、DROP(删除表)。
- 自动添加 LIMIT(限制返回数量)。
- 对时间范围和业务口径做确认。
- 执行前做语法和权限检查。

## 数据分析 Agent 角色拆分

即使只有一个 Agent，也可以在流程上拆成多个角色：

| 角色 | 职责 | 输出 |
| --- | --- | --- |
| 需求澄清器 | 确认指标、时间、维度 | 澄清问题或分析计划 |
| 数据定位器 | 找表、字段、权限 | 数据源清单 |
| 查询生成器 | 生成只读 SQL | SQL 草稿 |
| 查询审查器 | 检查 SQL 安全和口径 | 通过或驳回 |
| 分析执行器 | 聚合、计算、绘图 | 表格和图表 |
| 报告撰写器 | 生成业务解释 | 报告 |
| 质量审查器 | 检查幻觉和结论依据 | 修改建议 |

这样拆分有助于后续变成 LangGraph(图式智能体编排框架) 流程。

### 代码解释器

code interpreter(代码解释器) 用于执行受控代码，完成统计、绘图和文件处理。

适合任务：

- CSV 读取和清洗。
- 聚合统计。
- 相关性分析。
- 图表生成。
- 报告文件导出。

注意：代码解释器必须运行在 sandbox(沙箱) 中，限制文件、网络、时间和资源。

## SQL 生成安全策略

建议使用多层防护：

1. 只给模型提供允许访问的 schema(结构描述)。
2. 要求模型输出 SQL 和解释。
3. 使用 SQL parser(解析器) 做静态检查。
4. 禁止写操作和危险函数。
5. 添加默认 LIMIT(限制返回数量)。
6. 执行前检查用户权限。
7. 对大查询要求人工确认。

SQL 审查示例：

```text
检查项：
- 是否只读 SELECT
- 是否访问授权表
- 是否包含时间范围
- 是否有 LIMIT 或聚合
- 是否使用不存在字段
- 是否可能返回个人敏感信息
```

## 图表选择规则

Agent 生成图表时要选择合适图形：

| 分析目标 | 推荐图表 | 不建议 |
| --- | --- | --- |
| 时间趋势 | 折线图 | 饼图 |
| 类别对比 | 柱状图 | 复杂 3D 图 |
| 占比结构 | 条形图或饼图 | 折线图 |
| 分布情况 | 直方图、箱线图 | 单个均值 |
| 相关关系 | 散点图 | 双轴误导图 |

图表必须标注单位、时间范围和样本量。

### 报告生成

报告不应只是“漂亮总结”，而要包含：

- 问题定义。
- 数据范围。
- 核心发现。
- 图表。
- 可能原因。
- 不确定点。
- 建议动作。

## 分析结论分级

不要把所有分析都写成确定结论。推荐分级：

| 等级 | 含义 | 表达方式 |
| --- | --- | --- |
| 事实 | 数据直接支持 | “3 月销售额环比下降 12%” |
| 观察 | 从数据中看到的模式 | “下降主要集中在华东地区” |
| 假设 | 需要进一步验证 | “可能与渠道活动结束有关” |
| 建议 | 基于事实和假设提出动作 | “建议补充渠道投放数据验证” |

这种分级能显著降低“把相关性说成因果”的风险。

## 安全查询链路

推荐把 SQL 自动查询拆成多个步骤：

```text
用户问题
-> 识别业务指标
-> 匹配数据表和字段
-> 生成只读 SQL
-> 静态检查
-> 权限检查
-> 执行查询
-> 结果采样
-> 分析解释
```

静态检查示例：

```python
def validate_sql(sql: str) -> bool:
    forbidden = ["delete", "update", "drop", "insert", "alter"]
    normalized = sql.lower()
    return normalized.strip().startswith("select") and not any(
        word in normalized for word in forbidden
    )
```

## 最小实践

用一份销售 CSV 文件完成：

1. 读取 `date`、`city`、`amount`、`channel` 字段。
2. 计算每个城市本月销售额。
3. 对比上月增长率。
4. 输出增长最快的三个城市。
5. 生成一张柱状图。
6. 用自然语言解释结果。


## 评测样例

建议准备这些问题测试数据分析 Agent：

| 问题 | 重点考察 |
| --- | --- |
| 本月销售额是多少？ | 基础聚合 |
| 本月相比上月增长最快的城市？ | 时间对比和排序 |

## 可靠性设计

数据分析 Agent 最容易出错的地方：

| 风险 | 解决方式 |
| --- | --- |
| 指标口径错误 | 建立指标字典，关键指标需确认 |
| 字段猜错 | 只使用数据目录中存在的字段 |
| 查询过大 | 加 LIMIT、时间范围和资源预算 |
| 误读相关性 | 区分相关和因果 |
| 图表误导 | 展示单位、时间范围和样本量 |
| 数据权限越界 | 查询前做用户权限检查 |

## 常见误区

- 让模型直接写并执行任意 SQL。
- 没有指标口径，导致“销售额”含税、不含税、退款口径混乱。
- 只输出结论，不展示数据范围和计算方法。
- 把样本数据的相关性解释成确定因果。
- 没有保存分析过程，结果无法复现。
