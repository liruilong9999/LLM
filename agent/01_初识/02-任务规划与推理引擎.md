# 任务规划与推理引擎

## 本篇目标

本篇解决一个核心问题：Agent 如何从“用户给了一个模糊目标”，变成“一组可执行、可检查、可调整的步骤”。

学完后，你应该能：

- 解释 chain-of-thought(思维链) 和 tree-of-thought(思维树) 的用途与边界。
- 理解 ReAct(Reasoning and Acting，推理与行动) 循环。
- 设计一个简单的任务分解和动态调度流程。

## 先修知识

建议先读完 `01-Agent概念与架构总览.md`。你需要知道 Agent 至少包含目标、状态、工具和输出。

## 为什么需要规划

用户常常给的是目标，而不是步骤：

> “帮我分析这批客户投诉，找出主要原因，并生成一份改进建议。”

这个任务隐含了多个动作：

1. 读取投诉数据。
2. 清洗和分类。
3. 统计高频问题。
4. 结合业务背景解释原因。
5. 输出结构化建议。

如果没有规划，Agent 可能直接生成一段看似合理的总结，却没有真实依据。规划的价值是让“任务如何完成”变得可观察、可纠错。

## 三种常见推理方式

### 思维链

chain-of-thought(思维链) 指模型按步骤推理。工程上更常见的做法不是让模型暴露所有内部推理，而是要求它输出短计划、关键假设和可验证结论。

适用场景：

- 数学计算。
- 规则判断。
- 多条件筛选。
- 需要中间检查的问答。

实践建议：让模型输出“计划摘要”和“结果依据”，不要把不可控的长篇推理当成可靠证据。

### 思维树

tree-of-thought(思维树) 指同时探索多个候选路径，再选择更优路径。

适用场景：

- 方案设计。
- 复杂排障。
- 多策略搜索。
- 创意生成与评审。

简单例子：

```text
目标：降低客服 Agent 的错误回答率
路径 A：加强知识库检索
路径 B：增加回答前自检
路径 C：把高风险问题转人工
路径 D：引入离线评测集
```

工程上不一定真的生成树结构，但可以让 Agent 明确列出候选方案、优缺点和选择理由。

### ReAct 循环

ReAct 把推理和行动交替进行：

```mermaid
flowchart LR
    用户目标 --> 思考下一步
    思考下一步 --> 调用工具
    调用工具 --> 观察结果
    观察结果 --> 更新状态
    更新状态 --> 判断是否完成
    判断是否完成 -->|未完成| 思考下一步
    判断是否完成 -->|已完成| 输出答案
```

它的关键不是“模型想很多”，而是每一步都能把外部观察带回决策。

## 任务分解方法

一个实用的任务分解模板：

```text
目标：用户真正想完成什么
输入：已有信息是什么
缺口：还缺哪些信息
工具：哪些缺口可以通过工具补齐
步骤：先做什么，再做什么
停止条件：什么时候可以回答
风险：哪些动作需要确认或拒绝
```

例子：

```text
目标：判断今晚是否适合夜跑
输入：城市是上海，时间是今晚
缺口：天气、空气质量、降雨概率
工具：天气查询、空气质量查询
步骤：查天气 -> 查空气质量 -> 套用夜跑规则 -> 输出建议
停止条件：拿到天气和空气质量，或工具失败后请求用户稍后重试
风险：不能给医疗级建议
```

## 动态调度

动态调度指 Agent 不把计划写死，而是根据工具结果调整后续动作。

常见调度策略：

| 策略 | 适用场景 | 注意点 |
| --- | --- | --- |
| 顺序执行 | 步骤依赖明确 | 每一步都要检查失败 |
| 并行执行 | 多个工具互不依赖 | 需要合并结果和处理部分失败 |
| 条件分支 | 结果决定下一步 | 分支条件要清晰 |
| 人工确认 | 高风险操作 | 要展示动作、参数、影响范围 |
| 重试与降级 | 工具失败或限流 | 设置重试次数和备用方案 |

## 规划粒度

规划太粗会失控，规划太细会浪费。一个实用原则是：每一步都应该能被验证。

坏计划：

```text
1. 分析数据
2. 生成报告
```

这个计划看起来简单，但无法判断“分析数据”到底做了什么。

好计划：

```text
1. 确认数据表和字段是否存在
2. 统计本月和上月销售额
3. 计算城市维度增长率
4. 找出增长最快的三个城市
5. 检查样本量是否足够
6. 生成报告并标注数据范围
```

每一步都有输入、输出和检查方式。

## 状态驱动规划

复杂 Agent 不应该只依赖自然语言上下文，还应该维护结构化状态。

示例：

```json
{
  "goal": "生成销售下降原因分析",
  "known": {
    "time_range": "2026-03",
    "metric": "销售额"
  },
  "missing": ["数据表", "同比或环比口径"],
  "completed_steps": ["澄清指标口径"],
  "next_step": "查询数据目录",
  "risk_flags": ["可能涉及敏感销售数据"]
}
```

结构化状态的好处：

- 控制器能判断下一步。
- 日志能复盘执行过程。
- 失败后能从检查点恢复。
- 人工介入时能快速理解上下文。

## 计划生成与计划执行分离

建议把“生成计划”和“执行计划”分成两步。

```mermaid
flowchart LR
    用户目标 --> 计划生成
    计划生成 --> 计划校验
    计划校验 --> 执行器
    执行器 --> 工具调用
    工具调用 --> 结果观察
    结果观察 --> 计划更新
```

计划生成负责提出步骤，计划校验负责检查：

- 步骤是否和目标相关。
- 是否调用了不存在的工具。
- 是否遗漏必要输入。
- 是否包含高风险动作。
- 是否超过预算。

执行器只执行通过校验的步骤。这样可以避免模型临时生成危险动作。

## 停止条件设计

Agent 最怕无限循环。停止条件要具体：

| 停止类型 | 例子 |
| --- | --- |
| 任务完成 | 已获得所有必要证据并生成答案 |
| 信息不足 | 缺少城市、订单号、文件等关键输入 |
| 权限不足 | 用户无权访问数据 |
| 预算耗尽 | 达到最大模型调用或工具调用次数 |
| 风险触发 | 涉及资金、生产配置、法律承诺 |
| 连续失败 | 同一工具连续失败 2 次 |

伪代码：

```python
MAX_STEPS = 8

for step_index in range(MAX_STEPS):
    decision = plan_next_step(state)
    if decision.type == "final":
        return decision.answer
    if decision.type == "ask_user":
        return decision.question
    if is_high_risk(decision):
        return request_human_confirmation(decision)
    state = execute_and_update(decision, state)

return "任务步骤较多，我已经停止自动执行，请确认是否继续。"
```

## 常见任务规划模式

| 模式 | 流程 | 适用场景 |
| --- | --- | --- |
| 澄清优先 | 先问缺失信息，再执行 | 输入模糊、关键字段缺失 |
| 先检索后回答 | 搜索证据，再生成结论 | 知识库问答、政策解释 |
| 先计划后确认 | 生成计划，用户确认后执行 | 高风险工具调用 |
| 先小样本验证 | 先取少量数据验证口径 | 数据分析、大文件处理 |
| 先草稿后审核 | 生成草稿，审核后发布 | 邮件、公告、客服回复 |

## 最小实践

请为下面任务写一个 Agent 计划：

> “读取一份销售 CSV 文件，找出本月增长最快的三个城市，并生成简短报告。”

参考分解：

1. 检查文件是否存在、格式是否为 CSV(逗号分隔值文件)。
2. 读取字段并确认有日期、城市、销售额。
3. 按月份和城市聚合。
4. 计算增长率。
5. 生成前三名和解释。
6. 如果字段缺失，请用户补充或更换文件。

## 常见误区

- 让模型一次性完成所有步骤，导致错误不可定位。
- 把计划写得太细，增加 token(模型处理文本的基本单位) 成本和执行延迟。
- 没有停止条件，Agent 在工具调用和自我修正之间循环。
- 没有预算控制，复杂任务会消耗过多模型调用和外部 API。
