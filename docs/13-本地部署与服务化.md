# 13 本地部署与服务化

## 本章目标

这一章把“模型训练完或下载好以后，如何在本地真正跑起来”讲清楚。读完后你应该能理解：

- 什么叫本地部署
- 为什么量化（quantization，用更低精度表示权重）是本地运行的关键
- `transformers`、`llama.cpp`、`Ollama`、`vLLM` 分别适合什么场景
- 怎么从“本地能跑”走到“可以被程序调用的服务”

## 1. 什么叫部署

部署（deployment，把模型变成稳定可调用服务的过程）不只是“加载权重成功”，它至少涉及：

- 模型文件和 tokenizer 是否匹配
- 推理框架是否支持当前格式
- 硬件资源是否足够
- 响应延迟是否可接受
- 是否能被外部程序调用

## 2. 本地部署常见场景

### 场景 A：个人学习与实验

- 普通电脑
- 单用户
- 看重易用性

### 场景 B：本地业务工具

- 局域网服务
- 少量并发
- 更看重稳定性和维护便利

### 场景 C：高吞吐推理服务

- 多用户并发
- 更大显存和更专业的推理引擎

## 3. 为什么量化这么重要

量化会把模型权重从更高精度转成更低精度表示，例如：

- FP16（16 位浮点）
- INT8（8 位整数）
- 4-bit（4 比特量化）

### 工程意义

- 减少显存和内存占用
- 让更大的模型能在普通机器上运行
- 往往还能提升推理速度

$$
\text{模型内存占用} \approx \text{参数量} \times \text{每个参数的字节数}
$$

### 这个公式在算什么

它给你一个最基础的资源估算方法：模型需要多少内存或显存，首先取决于参数总数和每个参数占多少字节。

### 符号解释

- 参数量：模型里权重参数的总数
- 每个参数的字节数：例如 FP16 常见是 2 字节，INT8 常见是 1 字节

### 最小例子

一个 `7B`（70 亿参数）模型，如果按 FP16 粗略估算，权重本体约需要 `7e9 x 2` 字节，也就是十几 GB 级别；如果换成更低精度量化，占用会明显下降。

### 代价

- 可能带来精度损失
- 某些量化格式对框架支持有要求

## 4. 模型格式与工具链

### Hugging Face 原生格式

常见于 `transformers` 生态，适合：

- Python 内快速加载
- 训练后直接推理
- 与 `peft`、`datasets` 等生态组合

### GGUF

GGUF（适合 `llama.cpp` 生态的模型文件格式）常用于本地 CPU 或轻量 GPU 推理，尤其适合桌面端和边缘环境。

## 5. 用 `transformers` 本地推理

这是最容易理解、也最适合 Python 工程入门的路径：

```python
from transformers import AutoTokenizer, AutoModelForCausalLM

model_name = "Qwen/Qwen2.5-0.5B-Instruct"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)

inputs = tokenizer("请用中文解释什么是 Transformer", return_tensors="pt")
outputs = model.generate(**inputs, max_new_tokens=128)
print(tokenizer.decode(outputs[0], skip_special_tokens=True))
```

### 适合什么

- 学习
- 小型脚本
- 和训练代码无缝衔接

### 局限

- 高并发服务能力一般
- 对本地资源管理不如专门推理引擎

## 6. `llama.cpp` 适合什么

`llama.cpp` 是一个非常重要的本地推理项目，特点是：

- 对 CPU 推理支持好
- GGUF 生态成熟
- 很适合离线、本地、桌面场景

### 适合的读者

- 想在普通电脑上跑开源模型
- 想做离线工具
- 想深入理解量化和推理细节

## 7. `Ollama` 适合什么

`Ollama` 可以理解成一个更偏“本地模型运行时”的工具，特点是：

- 安装和运行简单
- 自带模型管理体验
- 对本地 API 调用友好

### 一个典型使用方式

```bash
ollama run llama3.2
```

或者通过 HTTP（超文本传输协议）接口调用。

### 适合什么

- 快速本地体验
- 做桌面或小型应用原型
- 想用最少配置把本地模型变成接口

## 8. `vLLM` 适合什么

`vLLM` 是更偏服务器和高吞吐推理的引擎，特点是：

- 对批处理和吞吐优化更强
- 对在线服务更友好
- 利用 KV Cache 管理和调度优化推理性能

### 适合什么

- 多用户并发服务
- 单机多请求高吞吐
- 更接近生产推理服务

### 局限

- 对环境要求更高
- 新手上手成本比 `Ollama` 更高

## 9. 怎么选工具

| 目标 | 推荐工具 |
| --- | --- |
| 想最快理解 Python 调用链 | `transformers` |
| 想在普通电脑上跑量化模型 | `llama.cpp` |
| 想快速拥有本地 API | `Ollama` |
| 想做高吞吐服务 | `vLLM` |

## 10. 一条现实可行的学习路径

如果你是新手，推荐按这个顺序实践：

1. 先用 `transformers` 在 Python 里跑通最小生成
2. 再用 `Ollama` 体验本地模型服务化
3. 然后尝试 `llama.cpp` 和 GGUF 量化模型
4. 最后再接触 `vLLM` 这类更偏服务优化的引擎

## 11. 从脚本到服务

你可以把最小服务理解成：

```mermaid
flowchart LR
    A["客户端"] --> B["HTTP API"]
    B --> C["推理服务"]
    C --> D["Tokenizer"]
    D --> E["模型生成"]
    E --> F["返回文本"]
```

### 最小 FastAPI（Python Web 框架）例子

```python
from fastapi import FastAPI
from pydantic import BaseModel
from transformers import AutoTokenizer, AutoModelForCausalLM

app = FastAPI()

tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-0.5B-Instruct")
model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-0.5B-Instruct")


class ChatRequest(BaseModel):
    prompt: str


@app.post("/generate")
def generate(req: ChatRequest):
    inputs = tokenizer(req.prompt, return_tensors="pt")
    outputs = model.generate(**inputs, max_new_tokens=128)
    text = tokenizer.decode(outputs[0], skip_special_tokens=True)
    return {"text": text}
```

这段代码不够工业级，但足够帮你理解“服务化到底是什么”。

## 12. 本地部署时最常见的问题

### 显存或内存不够

解决思路：

- 换更小模型
- 做量化
- 缩短上下文
- 使用 CPU 推理或更轻量框架

### tokenizer 和模型不匹配

表现为输出乱码、格式异常或效果明显不对。

### 生成太慢

可能原因：

- 模型太大
- 没有用 KV Cache
- 上下文太长
- 硬件不足

### 输出质量不稳定

可能原因：

- Prompt 模板不稳定
- 温度过高
- 模型本身和任务不匹配

## 13. 一个最小部署项目建议

项目：本地知识问答助手

最小功能：

- 本地加载开源指令模型
- 暴露 `/generate` 接口
- 提供简单前端或命令行调用
- 支持基础日志记录

如果继续升级，可以加入：

- RAG
- 多轮对话历史
- 会话缓存
- 访问控制
- 指标监控

## 14. 部署时的工程指标

除了“能跑”，还应关注：

- 首 token 延迟（first token latency）
- 每秒 token 数
- 峰值显存 / 内存
- 并发数
- 稳定性

$$
\text{总响应时延} \approx \text{Prefill 时延} + \text{Decode 单步时延} \times \text{生成 token 数}
$$

### 这个公式在算什么

它把一次响应粗略拆成两部分：先处理整段输入上下文的预填充阶段，再进行逐 token 解码生成阶段。

### 符号解释

- Prefill 时延：模型先读完整个输入 prompt 的耗时
- Decode 单步时延：每生成一个新 token 的平均耗时
- 生成 token 数：回答长度

### 最小例子

如果 prompt 很长，即使回答不长，Prefill 也可能成为瓶颈；如果回答很长，Decode 阶段会占主导。

这些指标决定一个系统到底“像实验”还是“像产品”。

## 常见误区

### 误区 1：部署就是把模型权重下载下来

不是。部署的核心是把模型变成稳定、可观测、可调用的系统能力。

### 误区 2：量化只会带来好处

不是。量化通常能省资源，但也可能带来精度损失和兼容性问题。

### 误区 3：`Ollama`、`llama.cpp`、`vLLM` 谁更先进

这个问题不够准确。它们面向的场景不同，关键是匹配你的目标。

## 面试可复述版

1. 本地部署的核心不是下载模型，而是把模型和 tokenizer、推理框架、硬件资源和服务接口整合起来。
2. 量化是本地运行大模型的关键手段，它能显著降低内存和显存占用，但可能带来精度损失。
3. `transformers` 适合 Python 端快速验证，`llama.cpp` 适合本地量化与 CPU 推理，`Ollama` 适合快速本地服务化，`vLLM` 更适合高吞吐服务场景。
4. 服务化通常需要把推理过程封装成 API，并关注延迟、吞吐、资源占用和稳定性。
5. 新手建议先从 `transformers` 或 `Ollama` 入手，再逐步过渡到更专业的推理引擎。
