# 14 面试与工程实战题

## 本章目标

这一章不是再讲新知识，而是帮你把前面 13 章的内容组织成“面试可说、项目可讲、工程可落地”的表达方式。重点包括：

- 高频原理题怎么答
- 工程选型题怎么答
- 排障题怎么答
- 项目经验怎么包装成清晰故事

## 1. 面试官通常在考什么

很多 LLM 面试题表面上问的是某个名词，实际上考的是四种能力：

1. 概念是否清楚
2. 原理是否能讲通
3. 工程取舍是否有判断
4. 你是否真的做过

所以答题最好按这个结构：

```mermaid
flowchart LR
    A["定义是什么"] --> B["为什么要这么做"]
    B --> C["怎么实现"]
    C --> D["优缺点与取舍"]
    D --> E["项目里怎么落地"]
```

## 2. 高频原理题

### 题 1：为什么 Transformer 能取代 RNN 成为主流

参考答法：

- RNN 依赖按时间步串行传递信息，训练难并行，长距离依赖也容易丢失。
- Transformer 用自注意力让任意两个位置直接建立联系，更容易建模长上下文。
- 它还能充分利用 GPU 并行能力，因此在大规模训练上更有优势。

### 题 2：为什么注意力公式要除以 $\sqrt{d_k}$

参考答法：

- Query 和 Key 做点积时，维度越大，数值越容易变大。
- 如果分数过大，softmax 会变得过于尖锐，梯度不稳定。
- 用 $\sqrt{d_k}$ 做缩放可以让数值范围更平稳。

### 题 3：BERT、GPT、T5 的区别

参考答法：

- BERT 是 Encoder-only，双向看上下文，偏理解任务。
- GPT 是 Decoder-only，左到右生成，偏文本生成和对话。
- T5 是 Encoder-Decoder，把任务统一成 text-to-text，偏条件生成。

### 题 4：LoRA 为什么省显存

参考答法：

- 因为它冻结原模型参数，只训练低秩增量矩阵。
- 可训练参数数量显著下降，优化器状态和梯度存储也跟着下降。

### 题 5：RAG 和微调怎么选

参考答法：

- 如果主要问题是知识更新和可追溯性，优先考虑 RAG。
- 如果主要问题是风格、格式、任务习惯，优先考虑微调。
- 真实项目里常常是两者结合。

## 3. 高频工程题

### 题 1：如何提升推理速度

回答思路：

- 用更合适的推理引擎
- 启用 KV Cache
- 缩短上下文长度
- 量化模型
- 选择更小模型
- 做批处理和并发优化

### 题 2：如何降低部署成本

回答思路：

- 从全精度改为量化模型
- 根据场景换成本地 CPU 推理或共享 GPU 服务
- 通过 Prompt / RAG 解决问题，避免盲目微调大模型
- 分层路由不同大小模型

### 题 3：模型回答经常幻觉怎么办

回答思路：

- 检查是否缺知识，若缺知识优先上 RAG
- 优化 Prompt，要求基于给定材料作答
- 降低温度
- 引入规则校验或人工审核
- 做专门的事实性评测集

## 4. 面试中常见评测公式

虽然面试题常常以口头形式出现，但很多岗位会追问你是否真的理解评测指标。最常见的是 Precision（精确率）、Recall（召回率）和 F1。

$$
\text{Precision} = \frac{TP}{TP + FP}
$$

$$
\text{Recall} = \frac{TP}{TP + FN}
$$

$$
F1 = \frac{2 \times \text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}
$$

### 这些公式在算什么

- Precision：模型给出的结果里，有多少是真的对的
- Recall：真实应该找出来的结果里，模型找出了多少
- F1：精确率和召回率的综合平衡指标

### 符号解释

- $TP$：真正例，预测对了且本来就应该为正
- $FP$：假正例，预测成正但其实不该为正
- $FN$：假负例，本该预测成正却漏掉了

### 最小例子

如果一个抽取任务中模型预测了 10 个实体，其中 8 个是对的，而真实应抽取 12 个实体，那么：

- Precision = 8 / 10 = 0.8
- Recall = 8 / 12 ≈ 0.667
- F1 会位于两者之间

## 5. 高频排障题

### 题 1：loss 降了，但效果没提升

可能原因：

- 训练数据和真实场景分布不一致
- 只优化了格式，不是真正优化业务目标
- 评测方法不对
- 过拟合训练集

### 题 2：服务变慢了

可能原因：

- 上下文变长
- 并发升高
- 没有正确使用 KV Cache
- 模型版本变大
- 部署框架配置不合理

### 题 3：模型输出乱码或风格异常

可能原因：

- tokenizer 和模型不匹配
- Prompt 模板损坏
- 解码参数异常
- 量化或加载过程有兼容问题

## 6. 项目表达模板

你可以用下面这个模板讲项目：

### 项目背景

我们要解决什么业务问题，为什么需要 LLM，而不是普通规则或小模型。

### 技术方案

模型底座是什么，是否用了 Prompt、RAG、LoRA、量化、本地部署。

### 关键取舍

为什么没做全参数微调，为什么用 RAG，为什么用某种部署框架。

### 效果指标

用了哪些离线和在线指标，最终提升了什么。

### 问题与优化

遇到什么问题，怎么定位，怎么优化。

## 7. 一个完整项目答法示例

### 示例：本地文档问答助手

可以这样讲：

1. 背景是团队内部有大量技术文档，成员检索成本高，传统关键词搜索召回质量差。
2. 我们采用本地开源指令模型加 RAG 的方案，而不是直接微调，因为文档更新频繁且需要引用出处。
3. 我们先对文档做清洗和 chunk，再用 embedding 模型建索引，在线阶段先检索再生成。
4. 为了降低部署成本，模型采用量化版本，本地通过 `Ollama` 或 `transformers` 服务化。
5. 评测上同时看命中文档准确率、回答正确率和人工满意度。
6. 后续针对幻觉问题，又加入了引用约束和低温度生成。

这个回答同时体现了：场景理解、架构选择、工程取舍和问题闭环。

## 8. 简历上怎么写更像真的做过

不要只写：

- 做了一个大模型项目

更好的写法是：

- 基于开源指令模型构建本地文档问答系统，采用 RAG 提升知识时效性，通过量化和本地服务化降低部署成本。
- 使用 LoRA 对模型进行领域表达适配，构建离线评测集并优化 Prompt 模板，提升格式稳定性与人工满意度。

### 为什么这样更好

因为它包含：

- 业务场景
- 技术方案
- 关键动作
- 结果方向

## 9. 面试时的表达原则

### 原则 1：先说结论，再讲原因

不要一上来铺太多背景。先给判断，再解释。

### 原则 2：讲取舍，而不是只讲名词

会背名词和会做工程差别很大。工程能力体现在“为什么这样选”。

### 原则 3：不会就说边界

如果你没做过 RLHF，就不要假装做过。可以诚实说：

- 我目前主要做过 SFT、LoRA、RAG 和本地部署
- RLHF 我重点理解过流程和论文，但没有完整上线实践

这种表达通常比模糊硬撑更可信。

3. 为什么 Decoder-only 成为现代通用 LLM 主流。

## 常见误区

### 误区 1：面试就是背八股

不是。LLM 面试越来越看重真实实践和取舍能力。

### 误区 2：项目讲得越复杂越好

不是。重点是逻辑清楚、边界清楚、问题清楚。

### 误区 3：必须做过超大模型训练才有资格面试

不是。对大多数岗位来说，更重要的是你是否理解原理、能实现最小闭环、能做工程落地。

## 面试可复述版

1. 回答 LLM 面试题时，最好按“定义、动机、实现、取舍、项目落地”来组织。
2. 常见高频题围绕 Transformer 原理、架构差异、微调方式、RAG 方案和部署优化展开。
3. 工程题更关心你如何做选型、如何定位问题、如何衡量效果。
4. 项目表达不能只说“用了某某模型”，而要说清目标、方案、取舍、指标和优化。
5. 即使没有做过超大规模训练，只要能从零实现关键模块并完成本地部署与项目闭环，同样具备很强说服力。
