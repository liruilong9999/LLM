# Agent(智能体) 开发框架（下）：生态与选型

## 本篇目标

本篇帮助你在不同 Agent 开发路线之间做选择。学完后，你应该能：

- 区分代码框架、多智能体框架、低代码平台和自研方案。
- 根据项目阶段、团队能力、合规要求选择合适工具。
- 避免“为了酷炫框架而复杂化系统”。

## 先修知识

建议先读完 `07-Agent开发框架（上）LangChain与LangGraph.md`。你需要知道简单 Agent 和复杂流程编排的差异。

## 生态分类

Agent 生态可以粗略分成四类：

| 类型 | 代表 | 适合场景 | 主要风险 |
| --- | --- | --- | --- |
| 代码框架 | LangChain、LangGraph | 工程可控、可扩展项目 | 学习成本和版本变化 |
| 多智能体框架 | CrewAI、AutoGen、MetaGPT | 角色协作、研究原型、复杂任务分工 | 成本高、调试难 |
| 低代码平台 | Dify、Coze 等 | 快速验证、运营配置、非研发参与 | 深度定制和治理边界 |
| 自研轻框架 | 内部 Agent runtime | 强业务约束、合规、安全、性能 | 重复造轮子和维护成本 |

## 选型维度

### 任务复杂度

任务越简单，越不需要重框架。

- 简单问答：LLM + 工具调用。
- 多步骤任务：LangChain Agent 或 LangGraph。
- 多角色协同：多智能体框架或自研编排。
- 企业流程：状态机、权限、审计、人工介入优先。

### 团队能力

如果团队以业务开发为主，低代码平台能快速起步。如果团队有后端、数据、平台能力，代码框架更容易做长期治理。

### 运行环境

企业内部系统通常关心：

- 私有化部署。
- 日志审计。
- 权限隔离。
- 数据不出域。
- 成本监控。
- 版本可回滚。

这些因素往往比框架热度更重要。

### 团队组织方式

不同团队适合不同路线：

| 团队形态 | 推荐路线 | 原因 |
| --- | --- | --- |
| 个人学习 | 手写循环 + 少量框架 | 理解底层更重要 |
| 小团队原型 | 低代码平台或 LangChain | 快速验证业务价值 |
| 后端团队 | LangGraph + 自研治理层 | 可接入已有权限和日志 |
| 数据团队 | 代码解释器 + SQL 网关 + 评测 | 重点在数据口径和安全查询 |
| 企业平台团队 | MCP + 模型网关 + 工作流平台 | 重点在标准化和复用 |

选型不是单纯技术问题，也是团队协作问题。

### 数据敏感度

数据越敏感，越要偏向可控方案。

| 数据类型 | 示例 | 选型建议 |
| --- | --- | --- |
| 公开数据 | 天气、公开文档 | 可用云服务和低代码平台 |
| 内部文档 | 制度、流程、会议纪要 | 需要权限过滤和日志 |
| 客户数据 | 订单、合同、工单 | 需要私有化或严格授权 |
| 高敏数据 | 金融、医疗、法律、生产配置 | 需要强审计、人工确认、沙箱 |

### 生命周期

项目阶段不同，最优选择也不同：

```text
验证问题 -> 验证价值 -> 小范围试点 -> 生产上线 -> 持续运营
```

早期重点是速度，后期重点是治理。不要用试验阶段的架构直接承载生产流量。

### 可观测性

Agent 系统必须能追踪：

- 用户输入。
- 模型调用。
- 工具参数。
- 工具返回。
- 中间状态。
- 最终输出。
- 用户反馈。

如果框架或平台不支持这些能力，上线后排障会非常困难。

## 框架选择建议

### 从 LangChain/LangGraph 开始

适合需要代码可控、逐步工程化的团队。

优点：

- 生态成熟。
- 工具、模型、结构化输出支持丰富。
- LangGraph 适合做复杂流程。

注意：

- 版本迭代快，依赖要锁定。
- 不要把框架抽象泄漏到业务核心。
- 关键流程要有测试和观测。

### 使用多智能体框架

适合任务天然有角色分工的场景，例如：

- 软件需求评审。
- 研究资料收集。
- 多专家讨论。
- 项目计划拆解。

使用前要问：

- 多 Agent 是否真的比单 Agent 更可靠？
- 角色之间如何共享上下文？
- 冲突结论由谁裁决？
- 成本和延迟是否可接受？

### 使用低代码平台

适合快速验证和业务配置：

- 内部知识库问答。
- 客服机器人原型。
- 固定流程的内容生成。
- 运营人员维护 prompt(提示词) 和知识库。

注意：

- 高度定制的工具链可能受限制。
- 复杂权限和审计要提前验证。
- 重要能力不能只存在平台配置里，要有导出和备份方案。

### 自研轻框架

适合业务约束强、合规要求高或已有平台能力的团队。

建议只自研这些稳定边界：

- 工具注册与权限。
- 状态存储与审计。
- 模型调用网关。
- 评测和回放。
- 人工确认与工单流转。

不要轻易自研模型适配、向量检索、可观测性等已有成熟方案，除非有明确约束。

## 选型评分表

可以给候选方案按 1 到 5 分打分：

| 维度 | 权重 | 说明 |
| --- | --- | --- |
| 开发速度 | 15% | 能否快速做出原型 |
| 可控性 | 20% | 流程、权限、状态是否可控 |
| 可观测性 | 15% | 是否能追踪调用和错误 |
| 扩展性 | 15% | 能否支持更多工具和场景 |
| 安全合规 | 20% | 是否支持权限、审计、私有化 |
| 团队熟悉度 | 10% | 团队能否维护 |
| 成本 | 5% | 模型、平台、运维成本 |

示例：

| 方案 | 开发速度 | 可控性 | 安全合规 | 综合判断 |
| --- | --- | --- | --- | --- |
| 低代码平台 | 高 | 中 | 中 | 适合验证 |
| LangChain/LangGraph | 中 | 高 | 高 | 适合工程项目 |
| 多智能体框架 | 中 | 中 | 中 | 适合研究和协作任务 |
| 全自研 | 低 | 高 | 高 | 适合强约束平台 |

## 迁移策略

很多项目会经历从低代码到代码框架的迁移。为了避免被锁死，早期就要保留这些资产：

- prompt(提示词) 模板导出。
- 知识库原始文档。
- 工具 API 定义。
- 评测样例。
- 用户反馈和失败样本。
- 业务流程图。

只要这些资产可迁移，底层平台更换的成本就会低很多。

## 选型决策表

| 场景 | 推荐路线 | 理由 |
| --- | --- | --- |
| 一周内做演示 | 低代码平台或 LangChain | 快速验证价值 |
| 内部工具助手 | LangChain + MCP | 接工具方便，工程成本低 |
| 复杂审批流程 | LangGraph 或业务状态机 | 状态和分支可控 |
| 多专家评审 | 多智能体框架 + 人工仲裁 | 角色分工清晰 |
| 企业客服上线 | 自研治理层 + 框架组件 | 权限、审计、评测更重要 |
| 高风险写操作 | 图编排 + 人工确认 | 需要可追踪和可拦截 |

## 最小实践

请为下面需求做选型：

> “公司希望做一个内部知识库助手，支持问制度、查流程、创建工单。第一阶段两周内上线给 30 人试用。”

推荐初版：

```text
低代码平台或 LangChain 快速搭建问答原型
MCP 或内部 API 接入工单创建工具
所有写操作进入人工确认
保留日志、反馈和失败样本
两周后根据使用数据决定是否迁移到更可控的代码架构
```

## 常见误区

- 按框架名气选型，而不是按任务复杂度和上线约束。
- 认为多智能体一定比单智能体强。
- 忽略模型调用成本，原型能跑但生产账单失控。
- 没有退出机制，被低代码平台配置锁死。
- 所有能力都自研，导致交付慢且质量不稳定。
